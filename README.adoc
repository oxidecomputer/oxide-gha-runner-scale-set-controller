= Oxide GitHub Actions Runner Scale Set Controller

Run GitHub Actions workflow jobs on ephemeral Oxide instances using the GitHub
Scale Set API.

== How it works

This program manages exactly one GitHub Actions scale set.

. On startup, the program looks for a scale set with the configured name in the
  configured runner group. It adopts a matching scale set and reconciles its
  labels, or creates the scale set when none exists. Adoption does not require
  the scale set to have been created by this program.

. The program opens a scale set listener session with GitHub. GitHub reports
  the number of assigned jobs and job lifecycle events. The desired capacity
  is `min(assigned_jobs + min_runners, max_runners)`, so `min_runners` provides
  warm, idle capacity to reduce job start latency.

. A reconciliation loop compares desired capacity with the scale set's Oxide
  resources. It runs when GitHub reports a change and periodically checks for
  resources that need cleanup.

. To add capacity, the program requests a just-in-time runner configuration from
  GitHub and creates an Oxide instance, boot disk, and network interface. The
  program uses cloud-init user data to write the just-in-time configuration to
  `/run/github-actions-runner/jitconfig` within the instance, which the instance
  must use to start the runner process. See <_runner_image, Runner image> below
  for more information. Each instance runs at most one workflow job.

. After the job finishes, the runner image must shut down the instance. The
  program removes the GitHub runner registration and deletes the instance and
  its boot disk. It can also stop and clean up instances during scale-down or
  when the expected runner lifecycle fails.

=== Stateless reconciliation and resource ownership

The program has no database or durable local state. It rebuilds its view by
listing instances and boot disks in the configured Oxide project and checking
the corresponding GitHub runner registrations. A restart resumes management of
existing resources as long as the GitHub configuration URL, scale set ID, and
Oxide project remain the same.

Oxide resources do not have tags, so ownership is encoded in their names. Every
runner name begins with `gha-runner-<hash>-`, where the hash is derived from the
canonical `github.config_url` and GitHub scale set ID. The GitHub runner, Oxide
instance, boot disk, and network interface use the same full runner name. The
program treats instances and disks with its prefix in the configured project as
owned resources. Do not use the same prefix for manually managed resources.

Changing `github.config_url`, `scale_set.name`, or `scale_set.runner_group`
selects a different GitHub scale set or ownership namespace. Changing the Oxide
project prevents the program from discovering resources in the old project.
Drain before making any of these changes.

IMPORTANT: Stateless reconciliation supports process restarts, not active-active
operation. Run only one active process for a given scale set. Use a separate
process and configuration file for each additional scale set.

== Configuration

See link:config.example.yaml[`config.example.yaml`] for example configuration.

=== GitHub authentication

`github.config_url` selects the GitHub repository, organization, or enterprise
where the runner scale set is registered. The program supports exactly one of
these authentication methods.

* *GitHub App* is preferred for repository and organization scale sets. Set
  `github.auth.app.client_id`, `installation_id`, and `private_key`. The private
  key can instead come from `GITHUB_APP_PRIVATE_KEY`. GitHub App authentication
  is not supported for enterprise-level runner scale sets.
* *Personal access token (PAT)* works for repository, organization, and
  enterprise scale sets. Set `github.auth.pat.token` or `GITHUB_TOKEN`. If
  `github.auth` is omitted and `GITHUB_TOKEN` is set, the program selects PAT
  authentication automatically. Enterprise-level scale sets require a classic
  PAT.

See the
https://docs.github.com/en/actions/how-tos/manage-runners/use-actions-runner-controller/authenticate-to-the-api[GitHub authentication documentation]
for the permissions required at each registration scope. Values in the YAML
configuration take precedence over environment variables. Prefer environment
variables or another secret-injection mechanism for the private key or token.

=== Oxide authentication

The program authenticates to an Oxide silo with an API token. Set the silo
URL and token with `oxide.host` and `oxide.token`, or use `OXIDE_HOST` and
`OXIDE_TOKEN`. YAML values take precedence over environment variables.

See the
https://docs.oxide.computer/api/guides/authentication[Oxide authentication]
for information on generating credentials. The program does not read the Oxide
CLI credentials file, so the host and API token must be supplied explicitly
through its configuration or environment.

The API token needs access to the configured project and permission to view
the image and manage instances and disks. Prefer environment variables or a
secret-injection mechanism over storing the token in `config.yaml`.

=== Runner image

The configured image must be prepared as an ephemeral, single-job
GitHub Actions runner image. This program does not install or start the
runner process, it only writes the just-in-time (JIT) configuration to
`/run/github-actions-runner/jitconfig` using cloud-init user data.

The runner and its dependencies intentionally belong to the image rather than
this program. Building them into the image makes instance startup
deterministic, avoids downloading executable code during provisioning, and
allows operators to control runner versions, workflow tools, patching, and
security policy.

The configured image is responsible for the following.

* Process cloud-init user data during boot and wait for it to complete before
  starting the runner process.
* Include the GitHub Actions runner, its system dependencies, and any tools
  required by workflows, such as Git or Docker.
* Start the runner process with the contents of
  `/run/github-actions-runner/jitconfig` passed to its `--jitconfig` option.
  Each runner must execute at most one workflow job.
* Shut down the instance whenever the runner process exits, whether the job
  succeeds or fails. The launcher must also shut down the instance if it
  encounters an error before or while starting the runner.
* Provide outbound HTTPS access to GitHub and any other services required by
  workflows.

This repository includes a link:packer[reference Packer configuration] for
building an Ubuntu 26.04 image that meets these requirements. Follow the
reference's build instructions, then set `scale_set.instance.image` to the
resulting Oxide image name or ID. Extend the package and provisioning steps when
workflows require additional tools.

Shutting down the instance signals that it is ready for cleanup. The program's
reconciliation loop removes the GitHub runner registration, instance, and boot
disk. If the image fails to start the runner or shut down correctly, periodic
reconciliation eventually reaps the resources, but this is a fallback rather
than the expected lifecycle.

== Development

Enter the Nix development environment, then use the Make targets to format,
lint, tidy dependencies, test, and build the project.

[source,console]
----
nix develop
make fmt
make lint
make tidy
make check
----

The `fmt`, `lint`, and `tidy` targets apply available fixes. `make check` checks
all three without changing files before running the tests and build.

== Usage

. Download a release binary from
  https://github.com/oxidecomputer/oxide-gha-runner-scale-set-controller/releases[GitHub releases].

. Copy `config.example.yaml` to `config.yaml` and replace the GitHub and Oxide
configuration with your own values.

. Start the application pointing to its configuration file.
+
[source,console]
----
oxide-gha-runner-scale-set-controller --config config.yaml
----

Send `SIGINT` or `SIGTERM` to stop the program.

=== Target the runners from a workflow

The example configuration includes `oxide-scale-set` as a runner label, so a
workflow can target the scale set by that label:

[source,yaml]
----
name: Oxide CI/CD Example
on: workflow_dispatch

jobs:
  hello:
    runs-on: oxide-scale-set
    steps:
      - run: uname -a
----

When `scale_set.labels` is omitted, the scale set name is its only label. When
labels are configured, they replace that default. Include the scale set name
in the labels if workflows should continue to use `runs-on: <scale-set-name>`.
Runner groups can further restrict which repositories and workflows may use the
scale set.

== Drain a scale set

Before changing the GitHub configuration URL, scale set name, runner group, or
Oxide project, stop the program and set both runner limits to zero:

[source,yaml]
----
scale_set:
  min_runners: 0
  max_runners: 0
----

Restart the program with the drain configuration. It stops acquiring jobs,
removes idle runners, waits for busy runners to finish, removes their Oxide
resources, deletes the GitHub scale set, and exits successfully. Apply the new
scope, scale set name, group, or project only after the drain completes.

= Oxide GitHub Actions Runner - Scale Set

Run GitHub Actions workflow jobs on ephemeral Oxide instances using the GitHub
Scale Set API.

== How it works

Each process manages exactly one GitHub Actions runner scale set:

. On startup, the controller looks for a scale set with the configured name in
  the configured runner group. It adopts a matching scale set and reconciles
  its labels, or creates the scale set when none exists. Adoption is based on
  the name and runner group; it does not require the scale set to have been
  created by this program.

. The controller opens a listener session with GitHub. GitHub reports the
  number of assigned jobs and job lifecycle events. The desired capacity is
  `min(assigned jobs + min_runners, max_runners)`, so `min_runners` provides
  warm, idle capacity in addition to current demand.

. A reconciliation loop compares desired capacity with the scale set's Oxide
  resources. It runs when GitHub reports a change, retries unfinished work, and
  periodically checks for resources that need cleanup.

. To add capacity, the controller requests a just-in-time runner registration
  from GitHub and creates an Oxide instance, boot disk, and network interface.
  Instance user data downloads and verifies the configured Linux x64 GitHub
  Actions runner, then starts it with the just-in-time registration. Each
  instance runs at most one workflow job.

. After the job finishes, the controller removes the GitHub runner
  registration, stops and deletes the Oxide instance, and deletes its boot
  disk. Idle instances are removed in the same way when demand decreases.

=== Stateless reconciliation and resource ownership

The controller has no database or durable local state. It rebuilds its view by
listing instances and boot disks in the configured Oxide project and checking
the corresponding GitHub runner registrations. A restart resumes management of
existing resources as long as the GitHub scope, scale set ID, and Oxide project
remain the same.

Oxide resources do not have tags, so ownership is encoded in their names. Every
runner name begins with `gha-runner-<hash>-`, where the hash is derived from the
canonical `github.config_url` and GitHub scale set ID. The GitHub runner, Oxide
instance, boot disk, and network interface use the same full runner name. The
controller treats instances and disks with its prefix in the configured project
as owned resources; do not use that prefix for manually managed resources.

Changing `github.config_url`, `scale_set.name`, or `scale_set.runner_group`
selects a different GitHub scale set or ownership namespace. Changing the Oxide
project prevents the controller from discovering resources in the old project.
Drain before making any of these changes.

IMPORTANT: Stateless reconciliation supports process restarts, not active-active
operation. Run only one active process for a given scale set. Use a separate
process and configuration file for each additional scale set.

== Configuration

See link:config.example.yaml[`config.example.yaml`] for example configuration.

=== GitHub authentication

`github.config_url` selects the GitHub repository, organization, or enterprise
where the runner scale set is registered. The controller supports exactly one
of these authentication methods:

* *GitHub App* is preferred for repository and organization scale sets. Set
  `github.auth.app.client_id`, `installation_id`, and `private_key`. The private
  key can instead come from `GITHUB_APP_PRIVATE_KEY`. GitHub App authentication
  is not supported for enterprise-level runner scale sets.
* *Personal access token (PAT)* works for repository, organization, and
  enterprise scale sets. Set `github.auth.pat.token` or `GITHUB_TOKEN`. If
  `github.auth` is omitted and `GITHUB_TOKEN` is set, the controller selects PAT
  authentication automatically. Enterprise-level scale sets require a classic
  PAT.

See the
https://docs.github.com/en/actions/how-tos/manage-runners/use-actions-runner-controller/authenticate-to-the-api[GitHub authentication documentation]
for the permissions required at each registration scope. Values in the YAML
configuration take precedence over environment variables. Prefer environment
variables or another secret-injection mechanism for the private key or token.

=== Oxide authentication

The controller authenticates to an Oxide silo with a device/API token. Set the
silo URL and token with `oxide.host` and `oxide.token`, or use `OXIDE_HOST` and
`OXIDE_TOKEN`. YAML values take precedence over environment variables.

Use `oxide auth login --host <silo-url>` to create a token with the Oxide CLI;
see the
https://docs.oxide.computer/api/guides/authentication[Oxide authentication]
for details. The controller does not read the Oxide CLI credentials file, so
the host and raw token must be supplied explicitly through its configuration or
environment.

The token's user needs access to the configured project and permission to view
the image and manage instances and disks. Prefer environment variables or a
secret-injection mechanism over storing the token in `config.yaml`.

=== Runner image

The configured image must be Linux x64, execute instance user data as `root`,
and provide `curl`, `sha256sum`, `tar`, `useradd`, `su`, `shutdown`, and the
GitHub Actions runner's system dependencies. It also needs outbound HTTPS access
to GitHub. Install any tools needed by workflows, such as Git or Docker, in the
image; the controller only installs the GitHub Actions runner itself.

== Usage

. Download a release binary from
  https://github.com/oxidecomputer/oxide-github-actions-runner-scaleset/releases[GitHub releases].

. Copy `config.example.yaml` to `config.yaml` and replace the GitHub and Oxide
configuration with your own values.

. Start the application pointing to its configuration file.
+
[source,console]
----
oxide-github-actions-runner-scaleset --config config.yaml
----

Send `SIGINT` or `SIGTERM` to stop the controller.

=== Target the runners from a workflow

The example includes `linux-x64` as a runner label, so a workflow can target the
scale set by that label:

[source,yaml]
----
name: Oxide runner example
on: workflow_dispatch

jobs:
  hello:
    runs-on: linux-x64
    steps:
      - run: uname -a
----

When `scale_set.labels` is omitted, the scale set name is its only label. When
labels are configured, they replace that default; include the scale set name in
the list if workflows should continue to use `runs-on: <scale-set-name>`.
Runner groups can further restrict which repositories and workflows may use the
scale set.

== Drain a scale set

Before changing the GitHub scope, scale set name, runner group, or Oxide
project, stop the controller and set both runner limits to zero:

[source,yaml]
----
scale_set:
  min_runners: 0
  max_runners: 0
----

Restart the controller with the drain configuration. It stops acquiring jobs,
removes idle runners, waits for busy runners to finish, removes their Oxide
resources, deletes the GitHub scale set, and exits successfully. Apply the new
scope, scale set name, group, or project only after the drain completes.
